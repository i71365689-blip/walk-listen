# Skill 太多也头疼：海量能力怎么加载

Skill 一个两个很好管。等到几十上百个，麻烦就来了：上下文塞不下，模型选错工具，延迟和费用一起涨。这不是「再买点 token」能糊弄过去的。

先认清问题：海量 Skill 的本质，是检索与调度问题，不是提示词写作问题。

常用几招。

一是路由。先用小分类器或轻量模型判断「这像哪一类任务」，再只加载那一小撮 Skill。像医院分诊，先挂号科别，再进诊室。

二是检索式加载。给每个 Skill 写短描述，做成可检索索引。用户意图来了，先找回最相关的几个，再展开完整说明书。

三是分层。目录层只告诉模型「我有支付、开票、退款」；真正执行退款时，才加载退款细则和危险操作确认。

四是预算。规定单次最多带几个 Skill、最多多少字说明。超了就压缩或降级，防止一次对话把窗口吃光。

五是治理。看哪些 Skill 长期没命中，该下线；哪些总冲突，该合并。没有观测，能力库会变成垃圾场。

面试可以这样答：Skill 规模化之后，要靠路由、检索、分层和预算做动态加载，并配合权限与版本管理。目标是让 Agent 在关键时刻只看见关键能力，而不是把整座军火库扛上街。

走着听完这条，你就懂了：Skill 多不可怕，不会选才可怕。
